§5.3 · Chain of Responsibility

Sistema de plugins por capas.

Enriquecer el esquema sin tocar el núcleo del generador.
Cadena de prioridad
10

20

30

40
WidgetPlugin x-widget
50
ReadonlyPlugin x-readonly
60
OrderPlugin x-order
Interfaz común
class SchemaPlugin(Protocol):
    priority: int

    def apply(
        self,
        schema: dict,
        model: type[Base],
        field_name: str,
        column_info,
    ) -> dict: ...
Por qué importa
Añadir un comportamiento nuevo es escribir apply() y registrar la prioridad. Cero cambios en el núcleo.

§5.5 — §6.2.4 · Operaciones en bloque

Crear, actualizar y eliminar en una sola petición.

POST /<entidad>/bulk · payload
[ { "data": { "name": "Nueva fila" } }, { "id": 42, "data": { "name": "Modificada" } }, { "_op": "delete", "id": 17 } ]
respuesta
{ "results": [ { "ok": true, "id": 101 }, { "ok": true, "id": 42 }, { "ok": false, "errors": [...] } ], "summary": { "created": 1, "updated": 1, "deleted": 0, "failed": 1 } }
Semántica transaccional · ?on_error=
partial
SAVEPOINTS SQL
Cada operación se aísla con begin_nested(). Un fallo no afecta a las exitosas.
abort
TRANSACCIÓN GLOBAL
Cualquier error provoca ROLLBACK total de la petición.
El cliente elige la semántica según el caso de uso: importación tolerante o atómica.

V · ConclusionesAportaciones
17
§8.1.1 · Aportaciones principales

Cuatro aportaciones que validan la hipótesis.

01
Reducción del tiempo de scaffolding
Una nueva entidad requiere únicamente el modelo SQLAlchemy y make_simple_rest_crud(). API y UI quedan disponibles de forma inmediata.
02
Contrato de datos unificado
JSON Schema como lenguaje común entre capas elimina la duplicación de definiciones y reduce la probabilidad de inconsistencias.
03
Extensibilidad sin acoplamiento
Plugins por prioridad y tokens opacos de inyección permiten adaptar el framework a requisitos específicos sin tocar el núcleo.
04
Bulk con semántica configurable
La dualidad partial / abort ofrece flexibilidad real para la importación masiva de datos.
